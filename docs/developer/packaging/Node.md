# Node.js Package Packaging Guide

This document explains how to add npm/Node.js packages to the nix-registry.

## When to Use

Use Node.js packages when:

- Adding npm packages as CLI tools
- The package is available on the npm registry
- You need Node.js-based command-line utilities

## How It Works

Packages are built the official nixpkgs way, with
[`buildNpmPackage`](https://nixos.org/manual/nixpkgs/stable/#javascript-buildNpmPackage).
(The previous `node2nix` toolchain was removed from nixpkgs as of 26.05.)

npm registry tarballs ship the package's **pre-built** output (`dist/`, `lib/`,
`esm/`, ...) but do **not** include a `package-lock.json`, which
`buildNpmPackage` requires. So for each package we vendor a prod-only lockfile
and let `buildNpmPackage` fetch the runtime dependencies reproducibly. Because
the tarball is already built, the build step is skipped — only runtime
dependencies are installed.

## File Structure

Node.js packages live in the `node/22/` directory:

```
node/22/
  node-packages.json          # Manifest: package list + pinned hashes (edit this)
  build-npm.nix               # buildNpmPackage wrapper (shared build logic)
  export.nix                  # Maps the manifest to a set of derivations
  pkgs/<attr>/package-lock.json  # Vendored prod-only lockfiles (generated)
```

### node-packages.json

The manifest. Each entry is keyed by its **Nix attribute name** (use
underscores, not hyphens) and pins everything needed to build reproducibly:

```json
{
  "typescript_json_schema": {
    "npm": "typescript-json-schema",
    "version": "0.67.1",
    "tarballHash": "sha256-...",
    "npmDepsHash": "sha256-..."
  }
}
```

- `npm` — the npm package name (scoped names like `@upstash/cli` are fine)
- `version` — the exact version to package
- `tarballHash` — SRI hash of the registry tarball (`src`)
- `npmDepsHash` — SRI hash of the fetched npm dependencies

`tarballHash` and `npmDepsHash` are filled in by `pls gen:node:22` (see below);
you only hand-edit `npm` and `version`.

### build-npm.nix

The shared `buildNpmPackage` wrapper. It fetches the registry tarball, copies
in the vendored lockfile, strips `devDependencies` and lifecycle `scripts`
(the tarball is already built), and installs only runtime dependencies.

### export.nix

Reads `node-packages.json` and maps each entry through `build-npm.nix`,
producing an attribute set keyed by the manifest keys. New manifest entries are
exposed automatically.

### pkgs/&lt;attr&gt;/package-lock.json

The vendored, prod-only lockfiles, generated by `pls gen:node:22`. Do not edit
by hand.

## Adding a New Package

### Step 1: Add to node-packages.json

Add an entry keyed by the Nix attribute name. Leave the hashes empty — they are
filled in by the generator:

```json
{
  "your_new_package": {
    "npm": "your-new-package",
    "version": "1.2.3",
    "tarballHash": "",
    "npmDepsHash": ""
  }
}
```

### Step 2: Generate Lockfile + Hashes

```bash
pls gen:node:22
```

This runs `scripts/local/gen-node.sh 22`, which for every manifest entry:

1. resolves the npm registry tarball and records its `tarballHash`,
2. strips `devDependencies` and generates a prod-only
   `node/22/pkgs/<attr>/package-lock.json`,
3. records the `npmDepsHash` via `prefetch-npm-deps`.

### Step 3: Build It

`export.nix` exposes the package automatically. Build and test:

```bash
nix build .#your_new_package
./result/bin/your-new-package --version
```

The binary name comes from the package's `package.json` `bin` field, not the
Nix attribute name.

### Step 4: Add to CI Verification

Add the package to `.github/workflows/ci.yaml`:

1. Add `.#your_new_package` to the `nix shell` command
2. Add `your-new-package --version` to the verification commands

See [CI Verification](./CIVerification.md) for detailed instructions.

## Upgrading a Package

1. Bump `version` in `node-packages.json`.
2. Run `pls gen:node:22` to refresh the lockfile and hashes.
3. Test with `nix build .#<attr>`.
4. Commit with `update(minor): upgrade <package>` (or the appropriate bump).

## Package Naming Conventions

Manifest keys (and therefore Nix attribute names) use underscores instead of
hyphens:

```
npm: action-docs            -> attr: action_docs
npm: typescript-json-schema -> attr: typescript_json_schema
```

To expose a tool under a custom name (e.g. `sg` for
`@kirinnee/semantic-generator`), just choose that as the manifest key — the
binary name still comes from the package's `bin` field.

## Troubleshooting

### Build fails running a `prepare`/build script (e.g. `tsc`)

The install hook runs `npm pack`, which executes the package's `prepare` script
even with `--ignore-scripts`. `build-npm.nix` strips `scripts` from
`package.json` to avoid rebuilding from source (the tarball already ships its
built output), so this should not happen. If a package genuinely needs a build
step, it is not a good fit for this pre-built approach.

### `ENOTCACHED` while fetching an unmet peer dependency

`build-npm.nix` passes `--legacy-peer-deps` so npm installs exactly what the
vendored lock pins without trying to fetch unmet peer deps (which aren't in the
offline cache). If a new package still fails, confirm its runtime dependencies
are all present in the generated lockfile.

### Hash mismatch after upgrading

Re-run `pls gen:node:22` — it recomputes both `tarballHash` and `npmDepsHash`
from the manifest's `version`.
